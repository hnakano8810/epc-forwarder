# クラウド型RFIDデータ連携プラットフォーム基本設計書 (Azure完全版)
## プロダクト名（仮称）: EPC Forwarder

## 1. 概要
本システムは、エッジ（モバイル）で読み取った大量のRFIDデータをリアルタイムに集約し、外部のあらゆるWebシステム（各種クラウドWMS、基幹システム、SaaS等）に「URLと認証ヘッダーを1本設定するだけ」でセキュアかつ確実に横流し（Webhook連携）するためのiPaaS（データ連携プラットフォーム）である。

最大の特徴として、**「EPC ＆ Mask（ビットマスク演算）による共通検索キー方式」**をインフラ層に実装することで、GS1標準（SGTIN-96）から各社独自のエンコードルール（独自コード）までをプログラムの改修なしで等価に扱い、超高速なSKU集約と外部連携を実現する。

---

## 2. 業務戦略（コンテキストの分離）
全業務を同一のストリーミング形式で処理しようとせず、業務特性（入出荷 vs 棚卸）に応じてクラウド（DB）側の処理戦略を明確に分ける。

### 2.1 入出荷業務（リアルタイムストリーミング型）
- **特性**: 伝票あたりのデータ量は少ない（数件〜数百件）。その場でSKU明細・総枚数を端末に表示でき、かつ「読み取ったタグが確実にクラウドへ届いたか」を**到達性検証**できることが価値。
- **データフロー**:
  1. ハンディ（エッジ）からタグを読んだ瞬間にMQTT（QoS 1）で1件ずつクラウドに即時送信（ストリーミング）。QoS 1により到達を担保する。
  2. 完了ボタン押下時、ハンディ側から「ユニークEPCの総件数（例: `total_count: 45`）」を乗せた完了イベントを送信する。重複読取はエッジ側でユニーク化した件数を送ることで、サーバ側UPSERT（ユニーク化）結果と整合させる。
  3. クラウド側でDBの実際の受信件数（ユニーク）と突合する。**これはシステムの到達性検証であって、業務的な誤出荷判定ではない**（中身の正否は件数では分からない）。完了イベントがストリーミング本体を追い越しうるため、突合時には数百ms〜数秒の確定（settle）猶予を設けるか、不一致時に再突合する。
  4. **リカバリ策**: 不一致＝通信での取りこぼし（システム都合）とみなし、端末へ「再読取のお願い」をフィードバックする。現場は何も気にせず再読取のトリガーを引く。DB側は `UPSERT` 処理により重複を自動スルー（上書き）し、最新状態へ収束させる。
  5. **再読取＝再転送**: 本システムは「転送屋」であるため、同一伝票番号の再読取・再オープンを許容し、確定すれば再度転送する（受信側は冪等性キーで重複排除）。

### 2.2 棚卸業務（実在庫積み上げ・セッション確定型）
- **特性**: データ量が膨大（数万〜数十万件）。現場の作業員は「正解（あるべき理論在庫）」を知らない。複数ロケ・複数端末・複数日にまたがる1つの「棚卸セッション（キャンペーン）」として扱う。
- **RFID特性への配慮**: 電波の回り込みにより、隣接棚や対象外の商品を拾う「過剰読取（幽霊読取）」が起こりうる。後勝ち上書きと（将来の）読取除外モードでこれを吸収する（→7.3）。
- **データフロー**:
  1. ハンディ（エッジ）は「棚番・エリアID」のコンテキストと共に、読んだタグをMQTTで積み上げ送信する。
  2. クラウドはEPCをキーに **後勝ち（最新のロケ＋時刻で上書き）UPSERT** で蓄積する。同一タグを別ロケで読み直したら、最後に読んだロケのものとして収束する（＝ロケ単位ではなくセッション全体で整合）。
  3. 棚ごとの「ロケーション区切りイベント」は**進捗マーカー**として扱い、転送トリガーにはしない（ロケ単位の連携は誤りを生むため）。
  4. **仮確定（締め切り前の任意/定期転送）**: 管理画面の手動操作または定期スケジュールで、現時点の後勝ち集約スナップショットを `is_final: false` ＋ `snapshot_version`（単調増加）付きで基幹へ転送できる。セッションは `open` のまま、確定までに何度でも送れる。
  5. **確定（締め切り）**: 全作業完了後、管理者がWeb管理画面で「棚卸締め切り」ボタンを押下。これをトリガーに最終スナップショットを `is_final: true` で1回転送し、セッションを確定する。
  6. **差異追及はコア対象外**: 理論在庫との突合（差異追及）は連携先（WMS/基幹）の責務とする。本システムは「確定済み・後勝ち解決済みの実績スナップショット（SKU＋数量＋ロケ）」をきれいに転送するところまでを担う。差異突合が必要な顧客には別プロダクト/有料オプションとして提供する（→7.2）。

### 2.3 端末（handy）へのフィードバック
本システムが端末へ返すのは、**読み取った実績だけから生成できる情報**に限る（伝票の期待明細＝基幹データは持たないため、誤出荷判定等は行わない、→7.3）。

- **表示内容**:
  - 入出荷: SKU別枚数 ＋ 総枚数
  - 棚卸: ロケ別のSKU明細
- **配送方式（ハイブリッド）**:
  - **プッシュ**: 到達性検証の不一致（再読取のお願い）など即時通知。IoT Hub の Cloud-to-Device メッセージ（端末は `devicebound` トピックを subscribe）。MQTT 5 なら Response Topic + Correlation Data。
  - **プル**: SKU明細・総枚数・ロケ別明細のオンデマンド取得。軽量HTTPクエリAPI（`GET /sessions/{id}/summary` 等）。
- **MQTTの前提**: MQTTはpub/sub（投げっぱなし）で同期レスポンスを持たない（QoSのACKは到達確認のみでデータを返さない）。よって端末フィードバックは上記の下り（downlink）チャネルによる**非同期フロー**となる。
- **スコープ外**: 誤出荷・期待明細との突合は、伝票（期待明細）＝基幹連携を前提とし顧客ごとに千差万別のため初期スコープ外（→7.3）。

## 3. コアロジック：EPC & Mask による共通検索キー方式

### 3.1 処理ロジック（インフラ層での共通化）
データ受信時、Azure Functions側で `EPC & Mask` のビットAND演算のみを実行し、シリアル番号が `0` にクリアされた一意の「検索キー」を動的に生成する。これをDB（商品マスタ）のプライマリキーとしてインデックス検索を行うことで、プログラム側に条件分岐（IF文）を発生させず、O(1)の超高速処理を実現する。

[読取EPC]      302DB42318A0038000001231  (SGTIN-96 実データ)
[Mask値]    &  FFFFFFFFFFFFFFC000000000  (シリアル部＝下位38bitのみを0クリアするマスク)
[検索キー]     302DB42318A0038000000000  (➔ この値で商品マスタを一発検索)

> **マスク厳密化**: SGTIN-96のシリアルは38bit固定のため、マスクは「上位58bit=1／下位38bit=0」とする。ビット数がずれると商品アイテムコードまで削れて別SKUが衝突するため、スキームごとにマスクを厳密に定義・テストする。

### 3.2 業界標準および可変長（SGTIN-96）の内包
SGTIN-96仕様におけるパーティション値（会社コードと商品アイテムコードの区切り位置の変動）の差異についても、商品特定に必要なビット領域（14〜95ビット目等）は固定であるため、同一の固定Mask値を通すだけで、特別な前処理なしで全く同一の検索キーへと自動的に収束・内包される。

### 3.3 独自コード・マルチマスク対応（考慮ケースAのクリア）【v1スコープ外／将来タスク】
同一テナント内で「標準タグ（SGTIN-96）」と「自社独自コードタグ」が混在、あるいは商品カテゴリごとにエンコードルールが異なるケースに対応するため、Mask定義を「Maskマスタ」として複数保持可能とする。

> **v1スコープ**: 初期リリースはSGTIN-96固定マスク1本のみ。複数Mask対応は将来タスクとする。実装時は性能維持のため「Maskを順次適用して試す」方式ではなく、**先頭のヘッダービット（例: `0x30`=SGTIN-96）によるルーティング方式**を採用し、O(1)を維持する。独自コードはヘッダーがGS1標準と衝突しうるため、スキーム識別子の一意化も併せて設計する。

### 3.4 マスクによるシリアル/付帯情報の欠落に関する注意
本方式はSKU集約のためシリアル部をゼロクリアするため、**シリアル部やタグのユーザメモリに載る消費期限・ロット番号等の付帯情報はマスク適用時点で失われる**。賞味/消費期限管理が要件となる案件（例: 食品スーパー、SGTIN-198運用）では、マスク適用前に当該情報を抽出し、SKU集約キーとは別経路でペイロードに保持・転送する設計とする。

## 4. システムアーキテクチャ（Azureサーバーレス構成）

### 4.1 構成コンポーネントとデータフロー
1. **エッジ層 (モバイルアプリ / RFIDコレクタ)**:
   - モバイル側は既存の **RFIDLibrary** を用いて自社開発する（特定端末・特定MQTT実装への制約なし）。読んだ生のEPCをそのまま送信する。
   - 通信プロトコル: `MQTT (QoS 1)` または `MQTT over WebSocket`。接続方式は自由に設計可能。
2. **インサイト/集約層 (Azure IoT Hub)**:
   - 端末からの超大量のMQTTメッセージ（QoS 1）をロスなく一瞬で受け止める。
   - デバイス認証・デバイスツイン（device twin）により、メッセージへテナント情報を付与し、マルチテナントのルーティング基盤とする。
   - **v1では IoT Hub 内蔵の Event Hubs 互換エンドポイントから直接 Azure Functions をトリガーし、独立した Event Hubs を省略してコスト・遅延・運用を軽量化する。**
3. **高スループット・バッファリング層 (Azure Event Hubs) ※将来オプション**:
   - 秒間数万件の超大量ストリーミングデータをミリ秒単位で受け止める分散ログ型メッセージングサービス（緩衝材）。
   - 「複数コンシューマ」「7日超のメッセージ保持」「パーティション制御」が必要になった本番フェーズで追加する。
4. **データ加工・配信エンジン (Azure Functions)**:
   - v1は **Consumption（従量）プラン** を採用（コールドスタートは許容、固定費ほぼ0）。低遅延・VNet閉域参照が必須になった本番フェーズで Premium を検討。
   - IoT Hub（内蔵EH互換エンドポイント）トリガーを利用。Batch window機能により「1秒間 または 100件」の単位でデータを配列（バルク）として一括取得（マイクロバッチ処理）。
   - DBから該当テナントの「Mask値」「宛先URL」「認証ヘッダー」をロード。ホットパス負荷軽減のため、テナント設定とシークレットはメモリにTTLキャッシュする。
   - 取得した生データに対して「EPC & Mask」のビット演算を施し、商品特定（SKU化）を行う。
5. **永続化層 (Azure SQL Database)**:
   - 採用DBは **Azure SQL Database** とする（関係モデルが業務に適合し、セッション×実績の集約に強いため）。
   - **設計思想（薄い層の定義）**: 本システムはシステム・オブ・レコードにはならない。恒久的な真実（理論在庫等）は持たず、**セッションが転送を終えるまでの一時状態のみ**を保持する。複数端末の読取をサーバ側でマージしてきれいなSKU集約を作るための作業状態であり、これは「転送の前処理」の範囲。
   - 保持データ:
     - マスタ系: テナント設定、Maskマスタ、商品マスタ。
     - `session(tenant, type[伝票/棚卸], status[open→finalized→forwarded→archived→purged], created_at, finalized_at, forwarded_at)`。
     - `reading(session_id, epc) PK, search_key(masked), location, read_at, device_id, excluded(論理削除フラグ)` ← 後勝ちUPSERT。集約時は `excluded` をスキップ。一意制約は (tenant, session) スコープ。
     - `snapshot(session_id, version, is_final, sent_at, status)` ← 転送（仮確定/確定）の監査・再送用ログ。
   - **理論在庫テーブルおよび差異突合JOINは持たない**（差異追及はコア対象外、→2.2/7.2）。

## 5. 外部システム連携（Webhook仕様）
連携先となるレガシーなシステム（SKU管理のWMS等）が改修なしでそのままデータを受け取れる仕様を提供する。

### 5.1 カスタム認証のサポート
管理画面から、連携先システムが要求する認証方式（API Key、OAuthトークン等）に合わせて、任意のHTTPヘッダーを自由に追加・設定できる設計とする。
- 例: `Authorization: Bearer <TOKEN>`、`X-API-KEY: <KEY>`
- 安全性を考慮し、接続文字列やAPIトークンなどの機密情報は **Azure Key Vault** に格納し、Azure Functionsからセキュアに参照する（高頻度参照のスロットリング回避のためメモリにTTLキャッシュ）。
- **SSRF対策**: 宛先URLはユーザー任意設定のため、プライベートIP帯やメタデータエンドポイント（`169.254.169.254`等）への送信をブロックする許可／拒否リストを設ける。
- **真正性**: 送信ペイロードにHMAC署名を付与し、受信側が送信元の真正性を検証できるオプションを提供する。

### 5.2 データの「自動集約（サマライズ）」モード
きれいな「SKU・数量」へ集約するWebhookは、**マイクロバッチ単位ではなく「セッション単位の論理的完了/確定」をトリガーに、DBから対象全件を読み出して1回だけ集約POSTする**（マイクロバッチ内の`GroupBy`では同一SKUが複数バッチに分かれ、断片的な数量POSTが多発するため）。転送単位はセッション（伝票確定／棚卸の仮確定・締め切り）であり、**ロケーション単位ではない**。集約はAzure Functionsのコード内（C#のLINQ等）でメモリ上に`GroupBy`を掛けて行い、相手側のシステム改修を発生させない。

- **変換イメージ**:
  `[{"tag_id": "EPC_A_001"}, {"tag_id": "EPC_A_002"}, {"tag_id": "EPC_B_001"}]`
  ⬇️
  `[{"sku": "ITEM-AAA", "quantity": 2}, {"sku": "ITEM-BBB", "quantity": 1}]`

#### 5.2.1 「転送」と「確定」の分離（棚卸の仮確定）
転送（POST）はセッション確定とは独立に複数回行える。棚卸では締め切り前に、現時点の後勝ち集約を **フルスナップショット**（SKU集約済みのため数十万タグでも数千件規模・冪等）で繰り返し送れる。

| 種別 | トリガー | ペイロード | セッション状態 |
|---|---|---|---|
| 仮確定スナップショット | 管理画面の手動送信 or 定期スケジュール | `is_final: false` ＋ `snapshot_version`（単調増加） | `open` のまま |
| 確定スナップショット | 締め切り（棚卸）／完了・件数突合（伝票） | `is_final: true` ＋ 最終版 | `finalized → forwarded` |

- 受信側は「最新 `snapshot_version` でフル置換（replace-by-snapshot）」する。`is_final` で最終版を判別。
- **前提**: 仮確定モードは受信側が replace-by-snapshot を許容できる場合に有効化する。append しか出来ないレガシー受信先では、確定（締め切り）1回のみとする出し分けとする。

### 5.3 流量制限（スロットリング）の自動化
相手のシステムが秒間大量のアクセスに耐えられない場合に備え、流量を抑えてWebhookを自動調整送信（スロットリング）する。`host.json` の `maxConcurrentCalls` やインスタンス数制限は **Function App全体（=全テナント共通）** に効くため、1社の遅い受信先が全テナントを巻き込む（ノイジーネイバー）。そのため **宛先（テナント）単位のレート制御（トークンバケット、または宛先別キュー）をアプリ層で実装する** ことを基本とする。

### 5.4 エラー救済とポイズンキュー（デッドレターキュー）
相手のサーバーがメンテナンス等で落ちていた（500エラーやタイムアウト）場合、Azure Functionsの標準リトライポリシー（指数バックオフ：間隔を徐々に広げて再試行）を実行。
規定回数すべて失敗したメッセージは自動的に **ポイズンキュー（Storage Queue等）** へ退避させ、管理画面に「送信失敗ログ」として表示。ユーザーが後から画面上で「再送ボタン」をポチッと押せる仕様とする。

- **冪等性**: IoT Hub/Event Hubsトリガーは at-least-once であり、バッチ内1件失敗時の再処理で成功済みメッセージが再送（重複POST）されうる。Webhookペイロードに **冪等性キー（idempotency-key）** を付与し、受信側が重複排除できるようにする。
- **配信段の分離（将来）**: 個別リトライ・ネイティブなデッドレター・順序保証が必要になった本番フェーズでは、配信段を **Azure Service Bus**（取込=IoT Hub／配信=Service Bus の2段構成）へ分離することを検討する。

## 6. データライフサイクルと保持ポリシー
本システムはシステム・オブ・レコードにならない（→4.1）。永続化データは**セッション・スコープの一時状態**であり、「役目の終わり＝転送の完了」で寿命が決まる。

- **状態遷移**: `open`（読取受付中。棚卸は仮確定転送を繰り返してよい） → `finalized`（確定） → `forwarded`（最終転送成功） → `archived`（生データを安価なストレージへ退避） → `purged`（ホットDBから削除）。
- **保持の起点**: `forwarded`（最終転送成功）後にテナント別TTLを起算。再送・監査の窓を確保した後にパージする。**仮確定転送中（`open`）はパージしない。**
- **アーカイブ**: 必要に応じ生の `reading` を Azure Blob Storage へ退避し、ホットの Azure SQL を軽量に保つ。集約済みスナップショットと `snapshot` ログは監査用に保持。
- **実行**: タイマートリガーの Azure Functions で定期パージ/アーカイブを行う。
- **PoC方針**: パージは無効（全データ保持）。保持窓・アーカイブ有無はテナント設定とし、本番フェーズで有効化する。
- **未確定セッションの扱い**: 長期間 `open` のまま放置されたセッションのタイムアウト/自動確定ポリシーは運用要件に応じて設定する。

## 7. ロードマップ＆ビジネス戦略（カスタマイズの分離）

### 7.1 ローンチ時（初期リリース：SGTIN-96特化）
- **対応範囲**: システム標準として、SGTIN-96用の固定Maskを1本だけ実装してリリース。
- **ターゲット**: 最もボリュームの多い、アパレルや小売・流通系の「標準GS1/EPCタグ」を使う顧客。初期の開発コストを最小限に抑えて素早くプロダクトを市場に投入（PoC）する。

### 7.2 独自コード対応・差異追及（マネタイズ・受託領域）
- **ビジネスモデル**: コアの「薄い転送層」から外した重い機能を、高単価の個別カスタマイズ/有料モジュールとして提供する。
- **独自コード**: 基幹システム側から送られてくる独自の商品コードを、クラウド側のDB登録時に「RFID用検索キー」に逆変換してマスタ登録する**「エンコード・プラグイン（ルール定義エンジン）」**を個別実装する。
- **差異追及（在庫突合）**: 理論在庫を持たない/突合できない連携先を持つ顧客向けに、理論在庫の取り込みと棚卸実績との差異突合を**別プロダクト/有料オプション**として提供する（コアには載せない）。
- システムのインフラコア層（Azure Functionsのビット演算部分）を1ミリも改造することなく、設定値（Mask値）の追加と、連携部分（プラグイン）の修正だけで高速に個別案件をこなせるため、極めて高い利益率を担保できる。

### 7.3 将来拡張（バックログ）
- **マルチMask／独自コードルーティング**: ヘッダービットによるスキーム判定でO(1)を維持（→3.3）。
- **SGTIN-198 対応**: 英数字シリアル（198bit）への対応。食品スーパー等での **消費期限・ロット管理** を含む（マスクで欠落する付帯情報の別経路保持を実装、→3.4）。
- **読取除外（削除）モード**: 棚卸での過剰読取（幽霊読取）を取り消す機能。`reading.excluded`（論理削除フラグ）を前提にデータモデルへフックは済（→4.1）。管理画面/ハンディからのEPC除外UIと操作監査を将来実装する。
- **期待明細の汎用正規化（誤出荷検知の汎用化）**: 伝票/出荷指示の「期待明細」を、商品マスタのように**正規化した汎用入力**として取り込めれば、個別の基幹連携を作り込まずに誤出荷・過不足検知を汎用機能として提供できる可能性。期待明細はセッション・スコープの取引データとして扱い、恒久マスタ（SoR）にはしない（→2.3）。
- **明細（シリアル粒度）転送モード**: 連携先によっては「SKU＋数量」への集約（→5.2）ではなく、**個体（シリアル）単位の `SKU ＋ シリアル` 明細**をそのまま受け取りたいケースがある（個品トレーサビリティ、シリアライゼーション、真贋・個体ライフサイクル管理等）。この場合は Mask は SKU 特定にのみ使い、**元のEPC（シリアル）はペイロードに保持**して個体明細として転送する（→3.4 の付帯情報保持と同方針）。集約/明細はテナント・連携先ごとに選択可能とする。
- **デバイスプロビジョニング**: Azure IoT Hub Device Provisioning Service (DPS) による多数端末の自動登録・認証（X.509 or SAS）とテナント割当。
- **Event Hubs 段の追加**: 複数コンシューマ／長期保持／パーティション制御が必要になった時点で導入（→4.1）。
- **Functions Premium プラン**: 低遅延・VNet閉域参照が必須になった本番フェーズで移行。
- **配信段のService Bus分離**: 個別リトライ・デッドレター・順序保証の強化（→5.4）。
