# 詳細設計：EPC & Mask 共通検索キー方式

対象: 受信EPCから「検索キー」を生成し商品マスタを O(1) 特定するコアロジック（基本設計3）。
v1スコープ: **SGTIN-96 固定マスク1本**。複数マスク/ヘッダールーティングは将来（§5）。

---

## 1. 原理

```
検索キー = EPC AND Mask
```
シリアル部をビットANDで 0 クリアし、商品（SKU）を一意に表す上位ビットだけを残す。これを `product(tenant_id, search_key)` のPKで引く。プログラム側にIF分岐を作らず O(1)。

---

## 2. SGTIN-96 のビットレイアウト

| フィールド | ビット位置(MSB=95) | 幅 | 内容 |
|---|---|---|---|
| Header | 95–88 | 8 | `0x30`（SGTIN-96固定） |
| Filter | 87–85 | 3 | 荷姿区分（item/case/pallet等。**商品識別＝GTINではない**） |
| Partition | 84–82 | 3 | 会社コード/商品アイテムコードの桁割り |
| GS1 Company Prefix + Item Reference | 81–38 | **44（合計固定）** | 会社コード＋商品アイテムコード（partitionで内訳が変動） |
| Serial | 37–0 | **38** | 個体シリアル番号 |

- 合計 8+3+3+44+38 = 96 bit。
- **重要**: Company Prefix と Item Reference の内訳はpartitionで変わるが、**合計44ビットは固定**。よって Serial は常に**下位38ビット固定**。
- → 「下位38ビットだけを0クリアする固定マスク」が、partition値に依らず全SGTIN-96で同一の検索キーへ収束する（基本設計3.2）。

---

## 3. マスク値（厳密定義）

- 上位 58 bit = 1、下位 38 bit（Serial）= 0。
- 12バイト16進:

```
Mask = 0xFFFFFFFFFFFFFFC000000000
```

検算: `14×F`(56bit=1) + `C`(=1100, 2bit=1) → 58bit=1、続く `9×0`(36bit) + Cの下位2bit = 38bit=0。

> **誤りに注意**: `0xFFFFFFFFFFFFFC0000000000`（基本設計の初版例）は下位**42**ビットを0にしてしまい、Item Referenceの下位4bitまで削れて別SKUが衝突する。必ず下位**38**ビットのみ。

### 3.1 Filter ビットの扱い（v1の決定）
- v1マスクは **Serialのみ**を0化し、**Filterビット(3)は検索キーに残す**。
- 帰結: 同一GTINでも荷姿（item/case）が異なれば検索キーが異なり、`product` の別行になる。荷姿を別SKUとして扱う運用では自然。
- 「荷姿を無視して同一SKUに寄せたい」要件が出た場合は、Filter(3bit)も0化する別マスクを定義（将来のマルチマスクで対応、§5）。

## 4. テストベクタ

実装の単体テスト（`EpcForwarder.Core.Tests`）に投入する基準値。

| # | 読取EPC (hex) | Mask | 検索キー (hex) | 説明 |
|---|---|---|---|---|
| 1 | `302DB42318A0038000001231` | 上記 | `302DB42318A0038000000000` | 基本ケース（基本設計の例） |
| 2 | `302DB42318A0038000009999` | 上記 | `302DB42318A0038000000000` | 同一商品・別シリアル → #1と同一キー |
| 3 | `302DB42318A00380000000001` | — | （24桁でない＝不正長） | 長さ検証で reject |

バイト単位のAND（#1）:
```
EPC : 30 2D B4 23 18 A0 03 80 00 00 12 31
Mask: FF FF FF FF FF FF FF C0 00 00 00 00
AND : 30 2D B4 23 18 A0 03 80 00 00 00 00  → 302DB42318A0038000000000
```
> 実装確定時にFilter違い／partition違い（partition 0〜6）の網羅ベクタを追加する。

## 5. ヘッダールーティング（将来：マルチマスク）

v1はSGTIN-96単一マスク。将来、独自コードやSGTIN-198が混在する場合:
- **Maskを順次適用して試す方式は採らない**（O(N)化・DB照合増で破綻）。
- 先頭の **Header バイト**でスキームを判定し、対応マスクを引く**ルーティングテーブル方式**とする（O(1)維持）。
  - 例: `0x30`→SGTIN-96 / `0x36`→SGTIN-198 / 独自レンジ→カスタムマスク。
- 独自コードはHeaderがGS1標準と衝突しうるため、テナント設定でスキーム識別子を一意化する（基本設計3.3）。

---

## 6. GTIN → 検索キー エンコーダ（商品マスタ投入用）

`product` のPKは「マスク済みEPC（検索キー）」。よって既知のGTIN/商品コードを**検索キーへ正変換してマスタ登録**する必要がある（基本設計の onboarding）。

### アルゴリズム（SGTIN-96）
1. 入力: GTIN-14、GS1 Company Prefix 桁数（GCP length）、Filter値。
2. GCP桁数から **partition値** と Company Prefix/Item Reference の桁・ビット幅を決定（GS1 partition表）。
3. Company Prefix と Item Reference（GTINのインジケータ＋アイテム部）をそれぞれ整数化し、規定ビット幅へ配置。
4. Header=`0x30`、Filter、Partition、上記、**Serial=0** を連結 → 96bit。
5. この値が、その商品の検索キー（= 実タグをマスクした結果と一致）。

> Serial=0で組み立てた値は、実運用タグ（Serial≠0）にマスクを掛けた検索キーと**ビット一致**する。これがマスク方式の整合性の核。

### partition表（参考）
| partition | CompanyPrefix bits / 桁 | ItemRef bits / 桁 |
|---|---|---|
| 0 | 40 / 12 | 4 / 1 |
| 1 | 37 / 11 | 7 / 2 |
| 2 | 34 / 10 | 10 / 3 |
| 3 | 30 / 9 | 14 / 4 |
| 4 | 27 / 8 | 17 / 5 |
| 5 | 24 / 7 | 20 / 6 |
| 6 | 20 / 6 | 24 / 7 |

（CompanyPrefix + ItemRef = 常に44bit）

---

## 7. raw モード（マスク非適用）

`destination.payload_mode='raw'`（商品マスタ無し）では、マスクもSKU解決も行わない。
- 検索キー生成をスキップし、`reading.search_key = NULL`。
- 重複排除・後勝ちは `epc` のみで行う。
- 連携ペイロードは生EPC（シリアル）のリスト（data-model §6.3）。

---

## 8. 実装メモ

- 演算は12バイト固定であれば `UInt64`×上位/下位 もしくは `Span<byte>` のバイト単位ANDで実装可（C#）。SGTIN-198(25バイト)将来対応を見越し、**`byte[]`/`Span<byte>` のバイト単位AND**を基本とする。
- 入力EPCの長さ・Header検証を front で行い、不正長/未対応スキームは「未知タグ」レーン（data-model §6.4）へ。
- マスク値はDB（`mask.mask_value`）からロードし、テナント設定としてTTLキャッシュ。
